# DB 스키마 — PostgreSQL + Chroma (초안)

> **근거**: [[DART공시통역_통합기획서]] §5.1 시스템 구성도(PostgreSQL: `users` · `portfolios` · `disclosures` · `notifications` · `analysis_results` / Chroma: 공시 임베딩), §6 하이브리드 RAG 단계.
> **전제**: Hibernate `ddl-auto: validate` — DDL은 **Flyway 마이그레이션으로만** 생성. 엔티티는 이 스키마를 검증만 한다. ([[CLAUDE]] §6-3)
> **보안**: 매수가/수량 등 금융 개인정보는 **AES-256 암호화 컬럼**으로 저장, 키는 별도 KMS. (통합기획서 §11.1)
>
> 본 문서는 **초안**이다. Stage 구분·RAG 방식 변동 시 함께 갱신한다(§6 변동 가능성 안내). 변경은 새 `V{n}__*.sql` 추가로만.

---

## 1. 전제 규칙 (Flyway + ddl-auto: validate)

- DDL 변경은 `backend/src/main/resources/db/migration/V{n}__{설명}.sql` 로만. 적용된 파일은 **불변**.
- `ddl-auto`는 `validate`(또는 `none`) 고정. `create/create-drop/update` 운영 사용 금지.
- 모든 PK는 `BIGINT GENERATED BY DEFAULT AS IDENTITY` (JPA `GenerationType.IDENTITY` 매핑).
- 시각 컬럼은 `TIMESTAMPTZ`, 날짜는 `DATE`. 금액/신뢰도 등 정밀 수치는 `NUMERIC`.
- 분류/등급/채널/상태 등 열거값은 `VARCHAR` + `CHECK` 제약 (JPA `@Enumerated(STRING)` 매핑).
- 암호화 컬럼은 `BYTEA` (애플리케이션단 AES-256-GCM, `IV ‖ ciphertext ‖ tag` 결합 저장). `_enc` 접미사 + `COMMENT` 명시.
- 개인정보 삭제권 보장을 위해 사용자 데이터는 **soft delete**(`deleted_at`) 우선.

### 매핑 힌트 (validate 통과 기준)

| PostgreSQL 타입 | Java/JPA 타입 |
|------|------|
| `BIGINT IDENTITY` | `Long` + `@GeneratedValue(IDENTITY)` |
| `VARCHAR(n)` + CHECK | `String` 또는 `enum` + `@Enumerated(STRING)` |
| `NUMERIC(p,s)` | `BigDecimal` |
| `BYTEA` | `byte[]` |
| `TIMESTAMPTZ` | `OffsetDateTime` / `Instant` |
| `JSONB` | `String`(+컨버터) 또는 매핑 라이브러리 |

## 2. ERD 개요

```
       ┌────────────────┐   N    1   ┌───────────────┐
       │  consent_logs  │───────────▶│    users      │ 회원·인증·BM등급·알림설정
       │ (동의 이력 누적 │            └──────┬────────┘ (V7: 빈도/필터/거래시간외)
       │  INSERT-only)  │      ┌────────────┼───────────────┬───────────────┐
       └────────────────┘      ↓            ↓               ↓               ↓
                       ┌────────────────┐ ┌────────────────┐ ┌──────────────┐
                       │  portfolios    │ │ notifications  │ │  feedbacks   │ 분석 평가
                       │  (매수가/수량   │ │  (채널/상태/    │ │ (유용/부정확  │ (재투표=UPDATE)
                       │   AES-256 암호) │ │   재시도)       │ │  user+analy   │
                       └──────┬─────────┘ └────┬───────────┘ └──────┬───────┘
                              │ stock_code      │ disclosure_id      │ analysis_id (N:1)
                              ↓                 ↓                    │
                       ┌────────────────┐   1        N    ┌──────────────────┐
                       │    stocks      │◀───────────────│   disclosures    │ 공시원문(rcept_no 멱등)
                       │ (종목 마스터)   │   corp_code     └────────┬─────────┘
                       └────────────────┘                          │ 1
                                                                   ↓ N
                                                         ┌────────────────────┐
                                                         │  analysis_results  │◀── feedbacks (N:1)
                                                         │  (분류/신뢰도/요약) │  Stage 2~5 분석결과
                                                         └────────────────────┘  (신뢰도 confidence 필수)

   [Chroma] disclosure_embeddings  ← Stage 3 RAG (disclosures.id 로 연결)
```

> `stocks`는 §5.1의 5개 핵심 테이블을 정규화하기 위한 **보조 마스터 테이블**(`portfolios`·`disclosures`가 공통 참조). 핵심 5개 테이블은 SSOT 근거 그대로다.
> `consent_logs`(users 1:N), `feedbacks`(users N:1 · analysis_results N:1)는 통합기획서 §4.6·§11.1·§13.3 요건으로 추가된 도메인이다(V8·V9).

## 3. PostgreSQL 테이블 스키마

### 3.1 `users` — 사용자·인증·BM 등급

회원가입/인증(Spring Security + JWT, OAuth2 Kakao/Google/Naver), 알림 설정, Free/Pro/Premium 등급 관리.

```sql
-- V1__create_users.sql
CREATE TABLE users (
    id                  BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email               VARCHAR(255) NOT NULL UNIQUE,
    password_hash       VARCHAR(255),                 -- OAuth 전용 계정은 NULL 허용
    oauth_provider      VARCHAR(10)  CHECK (oauth_provider IN ('KAKAO','GOOGLE','NAVER')),
    oauth_id            VARCHAR(255),
    nickname            VARCHAR(50)  NOT NULL,
    phone_number_enc    BYTEA,                         -- 알림톡 사용 시에만 수집(선택), AES-256
    tier                VARCHAR(10)  NOT NULL DEFAULT 'FREE'
                                     CHECK (tier IN ('FREE','PRO','PREMIUM')),
    tier_expires_at     TIMESTAMPTZ,                   -- 구독 만료(Free는 NULL)
    notify_channel      VARCHAR(10)  NOT NULL DEFAULT 'KAKAO'
                                     CHECK (notify_channel IN ('KAKAO','TELEGRAM','EMAIL')),
    notify_enabled      BOOLEAN      NOT NULL DEFAULT TRUE,
    terms_agreed_at     TIMESTAMPTZ  NOT NULL,         -- 필수: 서비스 약관
    privacy_agreed_at   TIMESTAMPTZ  NOT NULL,         -- 필수: 개인정보 처리방침
    marketing_agreed_at TIMESTAMPTZ,                   -- 선택: 광고성(친구톡 등)
    created_at          TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at          TIMESTAMPTZ  NOT NULL DEFAULT now(),
    deleted_at          TIMESTAMPTZ,                   -- soft delete(삭제권 보장)
    CONSTRAINT uq_users_oauth UNIQUE (oauth_provider, oauth_id)
);

COMMENT ON COLUMN users.phone_number_enc IS 'AES-256-GCM 암호화(IV‖ct‖tag). PII — 평문 로깅 금지';

CREATE INDEX idx_users_tier ON users (tier) WHERE deleted_at IS NULL;
```

> **선택/필수 동의 분리**: 약관·개인정보는 필수, 마케팅은 선택. (통합기획서 §11.1, §4 가입 흐름)

#### V7 추가 — 알림 빈도·필터·거래시간 외 (통합기획서 §4.3, §9.2)

> V1 정의는 불변이므로 신규 컬럼은 `V7__add_notification_settings_to_users.sql`로만 추가한다.

```sql
-- V7__add_notification_settings_to_users.sql (요지)
ALTER TABLE users
    ADD COLUMN notify_frequency   VARCHAR(10) NOT NULL DEFAULT 'INSTANT'
        CHECK (notify_frequency IN ('INSTANT','DAILY_1','DAILY_2','WEEKLY')),
    ADD COLUMN notify_type_filter VARCHAR(15) NOT NULL DEFAULT 'ALL'
        CHECK (notify_type_filter IN ('POSITIVE_ONLY','NEGATIVE_ONLY','ALL')),
    ADD COLUMN off_hours_allowed  BOOLEAN     NOT NULL DEFAULT TRUE;

CREATE INDEX idx_users_digest_target ON users (notify_frequency)
    WHERE deleted_at IS NULL AND notify_enabled = TRUE AND notify_frequency <> 'INSTANT';
```

| 컬럼 | 타입 | 의미 |
|------|------|------|
| `notify_frequency` | `VARCHAR(10)` CHECK | 즉시(`INSTANT`) / 일1회(`DAILY_1`) / 일2회(`DAILY_2`) / 주1회(`WEEKLY`). `DAILY_*`·`WEEKLY`는 다이제스트 배치 발송 |
| `notify_type_filter` | `VARCHAR(15)` CHECK | `POSITIVE_ONLY`(호재만) / `NEGATIVE_ONLY`(악재만) / `ALL`(모두) — `analysis_results.sentiment`(`POSITIVE`/`NEGATIVE`)와 접두사 정합 |
| `off_hours_allowed` | `BOOLEAN` | 거래시간 외(장 마감 후·주말·공휴일) 알림 허용 여부 |

- 기본값(`INSTANT`/`ALL`/`TRUE`)이 기존 사용자에게 자동 적용되어 **행동 변화 없음**.
- 부분 인덱스 `idx_users_digest_target`은 다이제스트 배치 대상(비-`INSTANT`·활성)만 색인 → 배치 스케줄러 조회 최적화.
- **사이드 이펙트**: 알림 디스패처에 빈도 분기(즉시 vs 다이제스트) 경로 신설 필요 → **후속 작업**(§7 하단 후속 이슈).

#### 동의 시각 컬럼 의미 재정의 (V8 `consent_logs` 도입에 따른)

`terms_agreed_at` / `privacy_agreed_at` / `marketing_agreed_at`는 V8 이후 **"최초 동의 시각"**의 의미로 재정의한다(컬럼은 그대로 유지). 정책 버전별 재동의·동의 이력의 **권위 있는 출처(SSOT)는 `consent_logs`**다. 법적 증명(특정 정책 버전 동의 여부)은 반드시 `consent_logs`에서 조회한다.

### 3.2 `stocks` — 종목 마스터 (보조)

DART 고유번호(`corp_code`)와 KRX 종목코드(`stock_code`)를 잇는 정규화 테이블. `portfolios`·`disclosures` 공통 참조.

```sql
-- V2__create_stocks.sql
CREATE TABLE stocks (
    stock_code  VARCHAR(6)   PRIMARY KEY,              -- KRX 종목코드(6자리)
    corp_code   VARCHAR(8)   NOT NULL UNIQUE,          -- DART 고유번호(8자리)
    corp_name   VARCHAR(100) NOT NULL,
    market      VARCHAR(10)  CHECK (market IN ('KOSPI','KOSDAQ','KONEX')),
    sector      VARCHAR(100),
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE INDEX idx_stocks_corp_name ON stocks (corp_name);
```

### 3.3 `portfolios` — 보유 종목 (매수가/수량 AES-256)

보유 종목 CRUD. Free 3개 / Pro·Premium 무제한. 매수가/수량은 **선택 입력 + AES-256 암호화**.

```sql
-- V3__create_portfolios.sql
CREATE TABLE portfolios (
    id                 BIGINT     GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id            BIGINT     NOT NULL REFERENCES users (id) ON DELETE CASCADE,
    stock_code         VARCHAR(6) NOT NULL REFERENCES stocks (stock_code),
    avg_buy_price_enc  BYTEA,                          -- 매수가(선택), AES-256
    quantity_enc       BYTEA,                          -- 수량(선택), AES-256
    memo               VARCHAR(200),
    created_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT uq_portfolio_user_stock UNIQUE (user_id, stock_code)
);

COMMENT ON COLUMN portfolios.avg_buy_price_enc IS 'AES-256-GCM 암호화 매수가. 평문 저장/로깅 금지(통합기획서 §11.1)';
COMMENT ON COLUMN portfolios.quantity_enc     IS 'AES-256-GCM 암호화 보유 수량. 평문 저장/로깅 금지';

CREATE INDEX idx_portfolios_user  ON portfolios (user_id);
CREATE INDEX idx_portfolios_stock ON portfolios (stock_code);  -- 공시→영향 사용자 역조회
```

> 매수가/수량은 암호화되어 DB 정렬·연산 불가 → 손익 계산은 **복호화 후 애플리케이션 계층**에서 수행. Free 3종목 제한은 애플리케이션단 검증(또는 트리거).

### 3.4 `disclosures` — 공시 원문 (rcept_no 멱등)

DART OpenAPI 1분 폴링 수집 결과. **접수번호 `rcept_no`로 멱등 처리**(중복 발송 금지, [[CLAUDE]] §4). Stage 1 룰 기반 추출 결과 포함.

```sql
-- V4__create_disclosures.sql
CREATE TABLE disclosures (
    id               BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    rcept_no         VARCHAR(14)  NOT NULL UNIQUE,     -- DART 접수번호(14자리) = 멱등 키
    corp_code        VARCHAR(8)   NOT NULL,            -- DART 고유번호
    stock_code       VARCHAR(6)   REFERENCES stocks (stock_code),  -- 비상장 등 NULL 허용
    corp_name        VARCHAR(100) NOT NULL,            -- DART 원본(LLM 변형 금지)
    report_nm        VARCHAR(255) NOT NULL,            -- 공시 제목(원본)
    rcept_dt         DATE         NOT NULL,            -- 접수일자
    disclosure_type  VARCHAR(50)  NOT NULL,            -- 공시 유형(80여 종, 룰 분류)
    content_text     TEXT,                             -- 본문 추출 텍스트(Stage 1)
    attachment_url   VARCHAR(500),                     -- 대용량 원문 첨부(S3 등) 참조
    collected_at     TIMESTAMPTZ  NOT NULL DEFAULT now(),
    created_at       TIMESTAMPTZ  NOT NULL DEFAULT now()
);

COMMENT ON COLUMN disclosures.corp_name IS 'DART 원본 인용 — LLM 변형 금지, 룰 기반(Stage 1) 추출';

CREATE INDEX idx_disclosures_corp     ON disclosures (corp_code, rcept_dt DESC);
CREATE INDEX idx_disclosures_stock    ON disclosures (stock_code, rcept_dt DESC);
CREATE INDEX idx_disclosures_type     ON disclosures (disclosure_type);
CREATE INDEX idx_disclosures_rcept_dt ON disclosures (rcept_dt DESC);
```

> 회사명/수치/날짜 등 **DART 원본 인용 필드는 룰 기반(Stage 1)으로만 추출**하고 LLM이 변형하지 않는다. 대용량 본문은 `attachment_url`로 외부 스토리지(S3) 참조 가능.

### 3.5 `analysis_results` — LLM 분석 (Stage 2~5)

공시별 분석 결과. **신뢰도 `confidence` 필수**, 낮으면 "판단 보류" 표시. 분류는 호재/중립/악재.

```sql
-- V5__create_analysis_results.sql
CREATE TABLE analysis_results (
    id                  BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    disclosure_id       BIGINT       NOT NULL REFERENCES disclosures (id) ON DELETE CASCADE,
    sentiment           VARCHAR(10)  NOT NULL
                                     CHECK (sentiment IN ('POSITIVE','NEUTRAL','NEGATIVE')),
    confidence          NUMERIC(4,3) NOT NULL CHECK (confidence >= 0 AND confidence <= 1),
    is_withheld         BOOLEAN      NOT NULL DEFAULT FALSE,   -- 신뢰도 낮음 → 판단 보류
    summary             TEXT         NOT NULL,                 -- 자연어 요약(3줄)
    expected_reaction   VARCHAR(20)  CHECK (expected_reaction IN ('UP','FLAT','DOWN')),  -- Stage 4
    rationale           TEXT,                                  -- 근거(과거 사례 종합)
    stage_reached       SMALLINT     NOT NULL CHECK (stage_reached BETWEEN 1 AND 5),
    stage_details       JSONB,                                 -- Stage별 중간 산출(스키마 파싱 후 저장)
    model_name          VARCHAR(50),                           -- 사용 LLM(미정, 런타임 기록)
    input_tokens        INTEGER,
    output_tokens       INTEGER,
    created_at          TIMESTAMPTZ  NOT NULL DEFAULT now(),
    CONSTRAINT uq_analysis_disclosure UNIQUE (disclosure_id)   -- 공시당 최종 분석 1건
);

COMMENT ON COLUMN analysis_results.confidence  IS 'LLM 신뢰도 0~1. 낮으면 is_withheld=TRUE(판단 보류)';
COMMENT ON COLUMN analysis_results.stage_details IS 'Stage 2~5 원응답 — 항상 스키마(record/Zod) 파싱 후 저장(환각 방지)';

CREATE INDEX idx_analysis_disclosure ON analysis_results (disclosure_id);
CREATE INDEX idx_analysis_sentiment  ON analysis_results (sentiment);
```

> **LLM 응답은 항상 스키마(Java record) 파싱 후 저장** — 환각 방지([[CLAUDE]] §6-6). 면책 문구·"부정확함" 신고 경로는 표시 계층에서 동반. 공시당 최종 1건 유지(`UNIQUE`), Stage별 중간값은 `stage_details` JSONB.

### 3.6 `notifications` — 알림 발송 이력

영향 사용자 조회 → 채널별 발송(카카오 1순위 → 텔레그램/이메일) → 결과 기록 + 재시도. **(user, disclosure, channel) 멱등**으로 중복 발송 차단.

```sql
-- V6__create_notifications.sql
CREATE TABLE notifications (
    id              BIGINT      GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id         BIGINT      NOT NULL REFERENCES users (id) ON DELETE CASCADE,
    disclosure_id   BIGINT      NOT NULL REFERENCES disclosures (id) ON DELETE CASCADE,
    channel         VARCHAR(10) NOT NULL CHECK (channel IN ('KAKAO','TELEGRAM','EMAIL')),
    status          VARCHAR(10) NOT NULL DEFAULT 'PENDING'
                                CHECK (status IN ('PENDING','SENT','FAILED','RETRYING')),
    retry_count     INTEGER     NOT NULL DEFAULT 0,
    sent_at         TIMESTAMPTZ,
    error_message   VARCHAR(500),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT uq_notification_dedup UNIQUE (user_id, disclosure_id, channel)
);

CREATE INDEX idx_notifications_user   ON notifications (user_id, created_at DESC);
CREATE INDEX idx_notifications_status ON notifications (status) WHERE status IN ('PENDING','RETRYING');
```

> **공시 중복 발송 금지**([[CLAUDE]] §4): 접수번호 멱등(`disclosures.rcept_no`) + 발송 멱등(`uq_notification_dedup`) 이중 방어. 재시도는 exponential backoff.

### 3.7 `consent_logs` — 동의 이력 (INSERT-only, 법적 증거)

약관·개인정보·면책·마케팅 동의를 **정책 버전별로 누적 보존**. `users.*_agreed_at`(시각 단일 컬럼)은 정책 갱신·재동의 시 이전 이력이 덮어써져 "사용자가 v1.2 정책에 동의했는가"를 증명할 수 없으므로 별도 이력 테이블로 분리. (통합기획서 §11.1 리스크1 면책 동의·리스크3 개인정보 동의)

```sql
-- V8__create_consent_logs.sql
CREATE TABLE consent_logs (
    id              BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id         BIGINT       NOT NULL REFERENCES users (id) ON DELETE RESTRICT,
    consent_type    VARCHAR(15)  NOT NULL
                                 CHECK (consent_type IN ('TERMS','PRIVACY','DISCLAIMER','MARKETING')),
    agreed          BOOLEAN      NOT NULL,
    policy_version  VARCHAR(20)  NOT NULL
                                 CHECK (policy_version ~ '^v[0-9]+\.[0-9]+$'),
    agreed_at       TIMESTAMPTZ  NOT NULL DEFAULT now()
);

COMMENT ON TABLE  consent_logs IS '동의 이력 누적 — INSERT-only. 최신 상태는 (user_id, consent_type) MAX(agreed_at). 법적 증거.';
COMMENT ON COLUMN consent_logs.user_id        IS 'ON DELETE RESTRICT — 사용자 탈퇴 후에도 보존(통합기획서 §11.1)';
COMMENT ON COLUMN consent_logs.policy_version IS '시맨틱 버전 v{major}.{minor}. 약관/개인정보/면책 각각 별도 버전. 본문은 legal/{type}_v{X.Y}.md';

CREATE INDEX idx_consent_logs_user_type
    ON consent_logs (user_id, consent_type, agreed_at DESC);
```

**운영 규칙 (코드에서 반드시 준수)**

- **INSERT-only** — UPDATE/DELETE 금지(불변 증거). 재동의·철회도 새 행 INSERT(`agreed=FALSE`로 철회 기록).
- "최신 동의 상태" 조회 = `(user_id, consent_type)` 윈도우의 `MAX(agreed_at)` 행. 인덱스 `idx_consent_logs_user_type`가 이 조회를 지원.
- `policy_version`은 **시맨틱 `v{major}.{minor}`** 고정(`CHECK (policy_version ~ '^v\d+\.\d+$')`). 약관·개인정보·면책 각각 별도 버전 관리.
- **약관 본문**은 DB에 넣지 않고 `legal/{consent_type}_v{X.Y}.md`(예: `legal/terms_v1.2.md`)로 Git 저장소에 버전별 보존하며, **머지 후 수정 금지**(본문 불변 → 동의 증거의 무결성). 메이저 버전 변경 시 재동의 요구 정책은 후속 이슈.
- `ON DELETE RESTRICT` — 사용자 탈퇴 후에도 동의 증거 보존(개인정보 보존 의무 vs 삭제권은 별도 운영 정책으로 조율).

### 3.8 `feedbacks` — 분석 피드백 (재투표=UPDATE, 데이터 해자)

분석 상세 화면의 "이 분석 어땠나요? 유용함/부정확함" 평가 누적. 정확도 KPI 검증과 도메인 특화 모델 학습 데이터의 원천. (통합기획서 §4.6 피드백 흐름, §8.3 정확도 90% KPI, §13.3 데이터 해자)

```sql
-- V9__create_feedbacks.sql
CREATE TABLE feedbacks (
    id           BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id      BIGINT       NOT NULL REFERENCES users (id)            ON DELETE CASCADE,
    analysis_id  BIGINT       NOT NULL REFERENCES analysis_results (id) ON DELETE CASCADE,
    verdict      VARCHAR(15)  NOT NULL CHECK (verdict IN ('USEFUL','INACCURATE')),
    reason       TEXT CHECK (length(reason) <= 2000),                                       -- 부정확 사유(선택)
    created_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),
    CONSTRAINT uq_feedbacks_user_analysis UNIQUE (user_id, analysis_id)
);

COMMENT ON TABLE  feedbacks IS '분석 정확도 모니터링·모델 개선의 원천(통합기획서 §4.6, §13.3 데이터 해자)';
COMMENT ON COLUMN feedbacks.verdict IS 'USEFUL/INACCURATE. 동일 user+analysis 재투표는 UPDATE(uq_feedbacks_user_analysis)';

CREATE INDEX idx_feedbacks_analysis_verdict ON feedbacks (analysis_id, verdict);
CREATE INDEX idx_feedbacks_created          ON feedbacks (created_at DESC);
```

**운영 규칙**

- 동일 사용자가 같은 분석에 **재투표하면 INSERT가 아니라 UPDATE**(`uq_feedbacks_user_analysis` 유니크 제약으로 강제, `created_at` 유지·`updated_at` 갱신).
- `reason`은 `INACCURATE`일 때만 선택 입력. 길이는 `TEXT`(무제한)이되 **애플리케이션 계층에서 입력 길이 캡** 권장(과대 입력·악용 방지).
- `INACCURATE` 누적이 임계치(기준 미정) 이상이면 해당 `analysis_results`를 비공개 처리하는 정책(§11.1 리스크2 회피)은 **이번 범위 밖 — 후속 이슈**.
- `analysis_id`는 `analysis_results.id` 참조(N:1). 공시당 분석은 1건(`uq_analysis_disclosure`)이므로 피드백은 사실상 공시 단위 평가와 동치.

## 4. Chroma 컬렉션 정의 (Stage 3 RAG)

> §6 Stage 3: 공시 본문 임베딩 → Chroma 유사 공시 검색(동일 회사+유형 최대 5건 / 동일 유형 타 회사 최대 5건) → KRX 5일 주가 반응 결합.

### `disclosure_embeddings`

| 항목 | 값 | 비고 |
|------|------|------|
| collection name | `disclosure_embeddings` | 공시 본문 임베딩 |
| distance metric | `cosine` | 텍스트 의미 유사도 |
| embedding dim | **모델 미정**(§6.3) | 예: `text-embedding-3-small` 1536 / 다국어 sentence-transformers 768 |
| document | 공시 본문(또는 청크) | Stage 1 `content_text` |
| id | `disclosures.rcept_no` 또는 `id` | PostgreSQL 행과 1:1 연결 |

**metadata 스키마** (§5.1 "메타데이터: 회사/유형/날짜")

```jsonc
{
  "disclosure_id":   123,            // PostgreSQL disclosures.id (역참조)
  "rcept_no":        "20260528000123",
  "corp_code":       "00126380",
  "stock_code":      "005930",
  "corp_name":       "삼성전자",
  "disclosure_type": "단일판매ㆍ공급계약체결",
  "rcept_dt":        "2026-05-28",
  "sentiment":       "POSITIVE"      // 분석 후 갱신(선택) — 필터 검색용
}
```

**검색 쿼리 패턴** (Stage 3)

```python
# 1) 동일 회사 + 동일 유형 (최대 5)
collection.query(query_embeddings=[emb], n_results=5,
    where={"$and": [{"corp_code": corp_code}, {"disclosure_type": dtype}]})

# 2) 동일 유형 다른 회사 (최대 5)
collection.query(query_embeddings=[emb], n_results=5,
    where={"$and": [{"disclosure_type": dtype}, {"corp_code": {"$ne": corp_code}}]})
```

> 임베딩 일괄 생성/백필은 `scripts/data_collection/` Python으로 수행(통합기획서 §5.4). **임베딩 모델·차원은 MVP 벤치마크 후 확정**(§6.3) — 확정 시 본 문서와 Stage 3 구현을 함께 갱신.
> 주가 5일 반응은 KRX 데이터(일 1회 배치)에서 조회해 검색 결과에 결합한다. 본 초안은 주가 시계열을 별도 테이블/외부 조회로 두고 핵심 5테이블에 포함하지 않았다(필요 시 `stock_prices` 보조 테이블 추가 검토).

## 5. Stage ↔ 데이터 계층 매핑

| Stage | 처리 | 읽기/쓰기 |
|-------|------|-----------|
| 1 룰 추출 | 메타·본문 추출, 유형 분류 | `disclosures` INSERT (rcept_no 멱등) |
| 2 LLM 1차 | 분류 + 3줄 요약 | `analysis_results` UPSERT (sentiment·confidence·summary) |
| 3 임베딩+RAG | 본문 임베딩, 유사 공시 검색 | Chroma `disclosure_embeddings` write/query + KRX 주가 결합 |
| 4 LLM 2차 | 최종 판단 + 예상 반응 | `analysis_results` 갱신 (expected_reaction·rationale·stage_reached=4) |
| 5 재무/업황 | Premium 재무·업황 분석 | `analysis_results.stage_details` 보강 (stage_reached=5) |
| 발송 | 영향 사용자 조회 → 채널 발송 | `portfolios`(stock_code 역조회) → `users`(빈도·필터·거래시간외 판단) → `notifications` INSERT |
| 사용자 피드백 | 상세 화면 → 유용함/부정확함 평가 | `feedbacks` INSERT/UPDATE (재투표 시 UPDATE) |
| 동의 | 가입·재동의 시 정책 버전별 기록 | `consent_logs` INSERT-only (최신 = MAX(agreed_at)) |

## 7. 보안·컴플라이언스 체크 (이 스키마 기준)

- [ ] `avg_buy_price_enc` · `quantity_enc` · `phone_number_enc` — AES-256-GCM, 키는 KMS 분리, 평문 로깅 금지
- [ ] `password_hash` — BCrypt/Argon2 (평문·단순 해시 금지)
- [ ] `confidence` 필수 + `is_withheld`로 판단 보류 — 신뢰도 없는 호재/악재 단정 금지
- [ ] `rcept_no` UNIQUE + `uq_notification_dedup` — 공시 중복 발송 방어
- [ ] `deleted_at` soft delete — 개인정보 삭제권 보장(개인정보보호법/GDPR)
- [ ] `consent_logs` INSERT-only, `MAX(agreed_at)`으로 최신 동의 조회 — UPDATE/DELETE 금지(불변 증거)
- [ ] 약관 본문은 `legal/{consent_type}_v{X.Y}.md`(예: `legal/terms_v1.2.md`)로 버전별 보존, 머지 후 수정 금지
- [ ] `consent_logs.user_id` `ON DELETE RESTRICT` — 탈퇴 후에도 동의 증거 보존(통합기획서 §11.1)
- [ ] `feedbacks` 누적 → 분석 정확도 모니터링·모델 개선 경로 확보(통합기획서 §13.3 데이터 해자)
- [ ] 통합 테스트는 **Testcontainers PostgreSQL**(Mock DB 금지, [[CLAUDE]] §6-6)
