# PromptGuard 법률용어 오탐 개선 Spec

> 상태: Draft → **Approved** (2026-07-03, dc-tech-review 승인)

## 배경 / 목적

`PromptGuard.FORBIDDEN_PATTERNS`가 자본시장법 투자 권유 표현을 차단하기 위해 **단순 `contains`**로 `매수`·`매도`·`추천` 등을 검사한다. `stage2-body-in-prompt` 이후 공시 **본문**이 LLM 입력에 들어가면서, 본문의 **법률용어**("주식**매수**청구권", "**매도**청구권", "자기주식**취득**" 등)가 권유 표현으로 **오탐** → 정상 분석이 `is_withheld=true`로 억울하게 판단보류되고 요인/해설이 미노출된다.

**실측(2026-07-03)**: 93707 현대지에프홀딩스 합병 공시 재분석 시 confidence=0.85(정상)인데도, key_points의 "주식**매수**청구권"이 `Pattern.compile("매수")`에 매칭 → withheld 강제 → stage_details 비어 목업 카드 미노출.

본문 투입으로 이 오탐이 **폭증**한다(합병·증자·자기주식 공시 등 법률용어 다수). 목적: **실제 권유 표현("매수 추천", "꼭 사세요")은 계속 차단하되, 사실·법률용어의 매수/매도는 통과**시켜 오탐을 줄인다.

- 페르소나: A·E(투자자) — 유효한 분석이 억울하게 "판단 보류"로 가려지는 문제 해소.
- BM 티어: Free(요인/해설이 Free) — 전 티어 품질 영향.

## 요구사항

- [ ] `FORBIDDEN_PATTERNS`를 **권유 맥락 패턴**으로 재정의 — bare `매수`/`매도`/`추천` 제거, 대신 권유 동사·문맥과 결합된 표현만 차단
- [ ] 실제 권유 표현 계속 차단: "매수/매도 추천", "매수/매도하세요", "지금 매수", "꼭 매수", "매수 권유", "사세요/파세요", "수익 보장", "확정 수익" 등
- [ ] 법률·사실 용어 통과: "주식매수청구권", "매도청구권", "자기주식취득", "매수세/매도세"(시황 서술) 등
- [ ] 회귀 방지 단위 테스트 — 오탐 케이스(법률용어) 통과 + 실권유 케이스 차단 양방향
- [ ] 2중 가드 유지 — 프롬프트 L1(Stage2PromptBuilder) + 후처리 L2(PromptGuard) 구조 불변

## 영향 범위 (조사 결과)

- 영향 레이어: **backend(analysis)** 단독. FE·DB·외부 계약 무변경.
- 영향 파일:
  - 수정: `analysis/services/PromptGuard.java`(FORBIDDEN_PATTERNS 재정의 — `containsForbidden`/`anyForbidden` 로직은 그대로 재사용)
  - 테스트: `analysis/PromptGuardTest.java`(법률용어 통과 + 실권유 차단 케이스 추가)
- **DB 변경**: 없음. (기존 오탐 withheld된 분석은 **재분석**해야 갱신 — 운영, stage2-body-in-prompt 재분석과 병행)
- **외부 계약**: 없음. is_withheld 산출 로직만 정밀화.

## 관련 패턴 / 과거 사례

- `PromptGuard` 주석에 이미 경고: *"단순 contains — 부정문/맥락 구분 못함. 매칭 시 보류 처리(과보수 OK)"*, *"FORBIDDEN_PATTERNS 추가는 운영 피드백 기반으로만(과보수 시 정상 응답까지 보류)"* — 본 Spec이 그 과보수를 실측 기반으로 완화.
- `stage2-body-in-prompt`(2026-07-03)가 본문 투입으로 이 오탐을 표면화 — 직접 후속.
- 자본시장법 리스크는 통합기획서 §11.1 — 차단 대상은 "투자 **권유/추천**"이지 "매수/매도"라는 **단어 자체**가 아님(핵심 통찰).

## 리스크 / 법적 검토

- **[핵심] 과소 차단 위험**: 패턴 완화로 신종 권유 표현이 빠져나갈 수 있음 → 보수적으로 "권유 동사(추천/권유/하세요/타이밍/기회/지금) + 매수/매도" 조합을 넓게 커버. 완화하되 실권유는 반드시 차단(테스트로 고정).
- **다층 방어 유지**: L2(PromptGuard) 외에도 L1(프롬프트 금지 지시) + confidence 판단보류 + 면책 문구 + 부정확 신고 경로가 있어, L2 정밀화가 단독 방어선이 아님.
- **기존 오탐 데이터**: 이미 withheld된 분석은 재분석 전까지 그대로 — 재분석 시 정상화(별도 운영).

## 권장 구현 방향

- **접근 A(권장): 권유 맥락 패턴화** — bare `매수`/`매도`/`추천` 삭제, 다음으로 대체:
  - `매수\s*추천`, `매도\s*추천`, `매수\s*권유`, `매도\s*권유`, `(지금|당장|꼭)\s*매수`, `매수\s*(하세요|하시길|타이밍|기회)`, `매도\s*(하세요|타이밍)`, 기존 `사세요`·`파세요`·`수익\s*보장`·`확정\s*수익`·`꼭\s*사` 유지.
  - 장점: 법률용어(매수청구권 등)는 매수 뒤에 청구권/청구 등이 와서 미매칭 → 자연 통과. 실권유는 동사/부사 결합으로 매칭.
- **접근 B: 화이트리스트 차감** — bare 패턴 유지 + 법률용어 목록 예외. 단점: 법률용어 전수 열거 불가·유지보수 부담. → 비권장.
- **트레이드오프**: A가 유지보수·정확도 모두 우위. 단 신종 권유 문구는 운영 피드백으로 패턴 추가(기존 정책 유지).

### 외부 계약 영향

### 리스크 & 법적 검토
- **[핵심] 과소 차단(자본시장법 §11.1)**: 패턴 완화로 신종 권유 표현이 빠질 수 있음 → 권유 동사/부사(추천·권유·하세요·타이밍·기회·지금·꼭) + 매수/매도 조합을 넓게 커버 + 실권유 차단을 카드 #2 테스트로 고정. 다층 방어(L1 프롬프트 + confidence 판단보류 + 면책 + 신고)로 L2 단독 의존 아님.
- **회귀 위험**: 기존 PromptGuardTest의 forbidden 케이스가 새 패턴과 정합하는지 확인(기존 케이스가 bare 매수/매도 의존이면 문구 조정).
