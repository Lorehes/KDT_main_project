# 🤖 우리 서비스, 공시를 어떻게 분석하나요? (비전공자용)

> 대상: 개발 배경이 없는 팀원
> 목적: "DB에 저장된 공시가 어떻게 AI를 거쳐 호재/악재 판정과 자연어 요약이 되는지" 한 번에 이해
> 사전 권장: [데이터수집_과정_쉽게_설명.md](데이터수집_과정_쉽게_설명.md) 먼저 읽기 (이 문서는 그 다음 단계)
> 한 줄 요약: **"수집 모듈이 '새 공시!' 신호를 보내면, AI가 그 공시를 읽고 '호재인지 악재인지 + 3줄 요약 + 자신감 점수'를 답하고, 우리는 그걸 검열·저장한다."**

---

## 0. 등장인물(시스템) 먼저 이해하기

| 이름 | 정체 | 비유 |
| ---- | ---- | ---- |
| **AnalysisOrchestrator** | "새 공시 떴다" 신호를 받아 분석을 지휘하는 **총감독** | 카톡 알림 받자마자 일을 배분하는 팀장 |
| **Stage2Analyzer** | 실제 분석 1건을 진행하는 **실무자** | 공시 한 건을 들고 AI에 물어보고 결과를 정리하는 사람 |
| **Stage2PromptBuilder** | AI에게 보낼 **질문지를 만드는 사람** | 시험 출제자 |
| **OllamaLlmClient** (qwen3:4b) | 실제로 답을 내는 **AI 모델** | 외부 분석가 (현재 우리 PC에서 로컬 실행 중) |
| **Stage2Output** | AI 답변을 담는 **양식(JSON)** | 정해진 답안지 양식 |
| **PromptGuard** | AI 답변에서 위험한 말을 걸러내는 **검열관·편집자** | 법무팀 + 교정 담당 |
| **AnalysisResult** (DB) | 최종 분석 결과를 보관하는 **DB 테이블** | 분석 결과 캐비닛 |

## 1. 큰 그림 — 6단계로 이해하기

```
[1] "새 공시!" 신호 수신  →  [2] 중복 체크 (이미 분석됐는지)
                                            ↓
                                  [3] AI에게 보낼 질문지 만들기
                                            ↓
                                  [4] AI 호출 (LLM이 답함)
                                            ↓
                                  [5] 답변 검열 (PromptGuard)
                                            ↓
                                  [6] DB 저장 + "분석 완료!" 신호 발사
                                            ↓
                                      알림 모듈로 인계
```

### 단계별 한 줄 설명

1. **이벤트 수신** — 수집 모듈이 발사한 "새 공시 떴어요(id=12345)" 신호를 잡는다.
2. **중복 체크** — 이미 분석한 공시면 **그냥 패스** (멱등성).
3. **프롬프트 작성** — "이 공시를 호재/중립/악재로 분류해줘"라는 한국어 질문지를 만든다.
4. **LLM 호출** — AI 모델(qwen3:4b)이 질문을 읽고 **JSON 양식**으로 답한다.
5. **검열(가드)** — 답변에 "꼭 사세요" 같은 위험 표현이 있으면 **판단 보류 처리**, 너무 길면 자른다.
6. **저장 + 다음 단계 알림** — 결과를 DB에 저장하고, "분석 완료(id=12345)" 신호를 알림 모듈에 발사한다.

## 2. 각 단계 자세히

### 🟦 1단계 — "새 공시!" 신호 받기 (이벤트 구독)

**왜 이렇게 만들었나요?**
수집 모듈과 분석 모듈을 **직접 연결**하면, 한쪽이 느려질 때 다른 쪽도 멈춥니다. 그래서 **이벤트(신호)** 방식을 씁니다.

- 수집 모듈: 공시 저장 → "새 공시!" 신호만 던지고 끝.
- 분석 모듈: 그 신호를 듣고 있다가 잡으면 **별도 스레드**에서 분석 시작.

> 💡 비유: 식당에서 주방장이 "음식 나왔어요!" 종을 울리면, 홀 직원이 듣고 가져가는 방식.
> 주방장은 누가 가져가는지 신경 안 쓰고 다음 요리에 집중할 수 있음.

> 📁 관련 파일: `backend/.../analysis/services/AnalysisOrchestrator.java`

---

### 🟦 2단계 — 중복 체크 (멱등성)

같은 공시를 두 번 분석하면:
- **돈 낭비** (AI 호출은 비용/시간 발생)
- **사용자에게 알림 2번 전송 사고**

그래서 분석 시작 전 **"이 공시 이미 분석돼 있어?"**를 먼저 묻습니다.
이미 있으면 → 아무것도 안 함 (skip).

> 📌 DB에 `UNIQUE(disclosure_id)` 제약이 있어서, 동시에 두 번 들어와도 한 번만 저장됩니다 (이중 방어).

---

### 🟦 3단계 — AI에게 보낼 질문지(프롬프트) 만들기

**프롬프트(prompt)란?**
AI에게 보내는 **지시문 + 질문**입니다. AI의 답변 품질은 **질문 품질에 90% 좌우**돼서, 이 단계가 굉장히 중요합니다.

우리 프롬프트는 다음 4가지를 강제합니다:

1. **역할 부여** — "당신은 한국 DART 공시 분석 어시스턴트입니다."
2. **자본시장법 가드** — "'매수/매도 추천', '꼭 사세요', '수익 보장' 같은 표현 금지."
3. **공시 데이터 삽입** — 회사명, 보고서 제목, 분류값, 접수일
4. **답변 형식 강제** — "반드시 이 JSON 양식으로만 답하세요. 다른 텍스트 추가 금지."

> 💡 왜 JSON으로 답하라고 하나요?
> 사람이 읽는 자연어로 답하면 **컴퓨터가 파싱하기 어렵고**, AI가 헛소리를 끼워 넣을 수 있습니다.
> JSON 양식을 강제하면 우리가 정확히 어떤 필드를 받을지 통제 가능 → **환각(hallucination) 위험 감소**.

> 📁 관련 파일: `backend/.../analysis/services/Stage2PromptBuilder.java`

---

### 🟦 4단계 — AI 호출 (LLM)

**우리가 쓰는 AI 모델**
- **qwen3:4b** — 알리바바가 공개한 오픈소스 모델 (4B = 40억 파라미터)
- **Ollama**라는 도구를 통해 **우리 PC에서 직접 실행** (외부 클라우드 비용 0원)
- MVP 단계용. 실서비스에선 OpenAI/Anthropic/Google 같은 클라우드 모델로 전환 예정.

**호출 시 핵심 옵션 3가지**:
- `temperature=0.2` — **창의성 낮춤** (분류 작업은 일관성이 중요). 1.0이면 매번 다른 답.
- `format=json` — **JSON으로만 응답** 강제
- `think=false` — qwen3의 "생각 모드" 끄기. 켜두면 JSON 응답이랑 충돌해서 빈 답을 줌 (실측 발견된 버그)

**안전 장치**:
- 1초 → 2초 → 4초 → 8초로 **재시도 3회** (네트워크 일시 장애 대비)
- 그래도 실패하면 → 조용히 skip (다른 공시 분석은 계속)

> 📁 관련 파일: `backend/.../infrastructure/llm/OllamaLlmClient.java`

---

### 🟦 5단계 — 답변 검열 (PromptGuard)

AI가 아무리 잘 시켜도 가끔 **사고**를 칩니다:
- 프롬프트에서 "추천 금지"라고 했는데도 "매수 추천합니다" 라고 답할 때가 있음
- 3줄 요약을 시켰는데 5줄 횡설수설할 때가 있음 (qwen3:4b 실측)
- 자신감이 0.3인데 단정적으로 "100% 호재" 라고 말할 때가 있음

그래서 **AI 답변을 사용자에게 보여주기 전 반드시 검열**합니다.

**검열 3종**:
1. **금지 키워드 매칭** → "추천/매수/매도/꼭 사세요/수익 보장" 등이 발견되면 → `withheld=true` (판단 보류 처리)
2. **신뢰도 임계** → confidence가 설정값(예: 0.5) 미만이면 → `withheld=true`
3. **길이 cap** → 240자 초과면 → 마지막 마침표 위치에서 자르고, 안 되면 "…" 부착

> 💡 `withheld=true`가 된 분석은 화면에 **"AI가 판단을 보류했어요"**로 표시됩니다.
> 잘못된 호재/악재 단정으로 사용자가 손실 보는 것 > 안 보여주는 것 → **0순위 회피 원칙**

> 📁 관련 파일: `backend/.../analysis/services/PromptGuard.java`

---

### 🟦 6단계 — 저장 + "분석 완료!" 신호 발사

검열을 통과한 결과를 DB의 `analysis_results` 테이블에 저장합니다. 저장 항목:
- `sentiment` — POSITIVE / NEUTRAL / NEGATIVE (호재 / 중립 / 악재)
- `confidence` — 0.0 ~ 1.0 (AI 자신감 점수)
- `withheld` — 판단 보류 여부 (true면 화면에서 호재/악재 숨김)
- `summary` — 한국어 3줄 요약
- `model_name` — 어떤 AI 모델로 분석했는지 (qwen3:4b 등)
- `stage_reached` — 어디까지 분석했는지 (현재는 2단계)

저장 끝나면 **"분석 완료(id=12345)" 이벤트 발사** → 알림 모듈(아직 개발 중)이 듣고 카카오 알림톡 발송.

## 3. 5단계(Stage) 분석 깊이 — 우리 비전

현재 구현된 건 **Stage 1 ~ 2**입니다. 전체 비전은 5단계.

| 단계 | 이름 | 무엇을 하는가 | 현재 상태 |
| ---- | ---- | ---- | ---- |
| **Stage 1** | 룰 기반 분류 | 제목 키워드로 유형 분류 (AI 안 씀) | ✅ 완료 |
| **Stage 2** | LLM 1차 분류 | AI로 호재/중립/악재 분류 + 요약 | ✅ 완료 (이 문서!) |
| **Stage 3** | RAG 검색 | 과거 유사 공시를 벡터 DB에서 찾아 참고 | 🚧 개발 예정 |
| **Stage 4** | LLM 최종 판단 | Stage 3 결과를 합쳐 정밀 분석 | 🚧 개발 예정 |
| **Stage 5** | 재무·업황 통합 | 회사 재무제표·업황까지 결합 | 🚧 개발 예정 |

> 📌 사용자 티어별 차등 적용:
> - **Free** = Stage 1~2 (지금까지)
> - **Pro** = Stage 1~4
> - **Premium** = Stage 1~5

## 4. 자주 나오는 용어 사전

| 용어 | 쉽게 말하면 |
| ---- | ---- |
| **LLM (Large Language Model)** | 거대 언어 모델 — ChatGPT, Claude 같은 종류의 AI |
| **프롬프트(Prompt)** | AI에게 보내는 지시문/질문 |
| **JSON 스키마** | 답변 양식 — "어떤 필드를 어떤 타입으로 답하라" 강제 |
| **환각(Hallucination)** | AI가 사실이 아닌 걸 그럴듯하게 지어내는 현상 |
| **신뢰도(Confidence)** | AI가 자기 답에 얼마나 자신 있는지 0~1로 표현 |
| **withheld (판단 보류)** | "확신 부족 / 위험 표현 감지 → 사용자에게 결론 숨김" |
| **Sentiment** | 감성 분류 — POSITIVE(호재) / NEUTRAL(중립) / NEGATIVE(악재) |
| **이벤트 기반** | 직접 호출 대신 신호를 던지고 받는 방식 (느슨한 결합) |
| **비동기(Async)** | "기다리지 않고 다른 일 먼저 하기" — 분석은 시간 걸리니 백그라운드로 |
| **멱등성(Idempotency)** | 여러 번 실행해도 결과가 1번과 같음 (중복 방지) |
| **Ollama** | 로컬에서 LLM을 돌리게 해주는 도구 (외부 비용 0) |
| **temperature** | AI의 창의성 다이얼. 낮을수록 일관성, 높을수록 다양성 |

### 📄 코드 ①  — 신호 받고 분석 시작하는 총감독

📁 `backend/src/main/java/com/dartcommons/analysis/services/AnalysisOrchestrator.java`

```java
    @Async("analysisExecutor")
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void onDisclosureCollected(DisclosureCollectedEvent event) {
        Long disclosureId = event.disclosureId();
        try {
            stage2Analyzer.analyze(disclosureId).ifPresent(this::publishCompleted);
        } catch (Exception e) {
            log.warn("AnalysisOrchestrator: 예상치 못한 예외 ...", e);
        }
    }
```

🔎 **해석**
- `@TransactionalEventListener(AFTER_COMMIT)` → "수집이 DB에 **완전히 저장된 후에만** 이 메서드 실행"
  - 만약 수집이 중간에 롤백되면 분석은 시작도 안 함 → 유령 분석 방지
- `@Async("analysisExecutor")` → **별도 스레드 풀**에서 비동기 실행
  - 분석은 수 초~수 분 걸리므로, 수집 모듈을 기다리게 하면 폴링이 느려짐
- `stage2Analyzer.analyze(...)` → 실제 분석 실무자에게 위임
- `.ifPresent(this::publishCompleted)` → **분석이 성공했을 때만** "분석 완료" 이벤트 발사

### 📄 코드 ④  — LLM 호출 (OllamaLlmClient)

📁 `backend/src/main/java/com/dartcommons/infrastructure/llm/OllamaLlmClient.java`

```java
    @Retryable(
            retryFor = RestClientException.class,
            maxAttempts = 3,
            backoff = @Backoff(delay = 1000, multiplier = 2.0, maxDelay = 8_000)
    )
    public Stage2Output classifyStage2(String prompt) {
        Map<String, Object> body = Map.of(
                "model", props.model(),
                "prompt", prompt,
                "format", "json",
                "stream", false,
                "think", false,
                "options", Map.of(
                        "temperature", 0.2,
                        "num_predict", 400
                )
        );
        // ... HTTP POST 호출 + JSON 파싱 ...
    }
```

🔎 **해석**
- `@Retryable` → 실패 시 **3회까지 자동 재시도** (1초 → 2초 → 4초로 간격 증가 = 지수 백오프)
- `model` → `qwen3:4b` (또는 application.yml에 설정된 모델)
- `format=json` → **답변을 JSON으로만** 받기
- `stream=false` → 한 번에 전체 답변 받기 (스트리밍 안 함)
- `think=false` → **qwen3의 "생각 모드" 끄기** (켜두면 JSON과 충돌해 빈 응답 발생 — 실측 발견)
- `temperature=0.2` → **낮게 = 일관성 있는 분류** (1.0이면 매번 다른 답)
- `num_predict=400` → 최대 400 토큰까지 생성 (3줄 요약 + JSON 키 분량 여유)

> 💡 운영 중 실제 발견한 함정 2가지:
> - **qwen3 think 모드 함정** — 켜두면 JSON 응답 무한 thinking → 빈 응답. 반드시 `think=false`.
> - **DART 페이지 무한 루프** (수집 모듈) — 외부 시스템은 항상 의심해야 함.

### 📄 코드 ⑤  — 검열관 (PromptGuard)

📁 `backend/src/main/java/com/dartcommons/analysis/services/PromptGuard.java`

```java
    public static final int SUMMARY_MAX_CHARS = 240;

    private static final List<Pattern> FORBIDDEN_PATTERNS = List.of(
            Pattern.compile("추천"),
            Pattern.compile("매수"),
            Pattern.compile("매도"),
            Pattern.compile("사세요"),
            Pattern.compile("파세요"),
            Pattern.compile("수익\\s*보장"),
            Pattern.compile("손실\\s*없"),
            Pattern.compile("확정\\s*수익"),
            Pattern.compile("꼭\\s*사")
    );
```

🔎 **해석**
- 자본시장법상 위험한 표현들의 **블랙리스트**
- `\\s*`는 "공백 0개 이상" → "수익보장" / "수익 보장" / "수익  보장" 모두 잡힘
- 단순 contains 방식 — "이런 표현 절대 통과 안 시킴" 보수적 접근

---

```java
    public GuardResult sanitize(Stage2Output raw, double confidenceThreshold) {
        String summary = raw.summary() == null ? "" : raw.summary();
        boolean forbiddenHit = containsForbidden(summary);
        String capped = capLength(summary);

        boolean withheldByThreshold = raw.confidence().doubleValue() < confidenceThreshold;
        boolean withheld = forbiddenHit || withheldByThreshold;

        Stage2Output sanitized = new Stage2Output(raw.sentiment(), raw.confidence(), capped);
        return new GuardResult(sanitized, withheld, forbiddenHit, withheldByThreshold);
    }
```

🔎 **해석**
- `forbiddenHit` = 요약문에 금지 키워드 있어?
- `capped` = 240자 넘으면 자르기 (마침표 위치에서 자연스럽게)
- `withheldByThreshold` = 신뢰도가 설정값(예: 0.5) 미만이야?
- `withheld = forbiddenHit || withheldByThreshold` → **둘 중 하나라도 걸리면 판단 보류**
- 원본 `raw`는 건드리지 않고 **새 객체** `sanitized`를 만들어 반환 (불변 원칙)

> 💡 검열의 우선순위: **잘못된 판단으로 사용자가 손실 = 0순위 회피**.
> 보류로 인한 "정보 못 봄"의 비용 < 잘못된 호재/악재 단정의 비용.
