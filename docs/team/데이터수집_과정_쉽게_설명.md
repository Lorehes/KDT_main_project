# 📊 우리 서비스, 데이터를 어떻게 모으나요? (비전공자용)

> 대상: 개발 배경이 없는 팀원
> 목적: "공시 데이터가 어떻게 들어와서 분석 직전까지 가는지" 한 번에 이해
> 한 줄 요약: **"DART 사이트(정부 공시 창고)에서 우리가 관심 있는 회사의 새 공시만 골라, 1분마다 가져와 우리 DB에 저장한다."**

---

## 0. 등장인물(시스템) 먼저 이해하기

| 이름 | 정체 | 비유 |
| ---- | ---- | ---- |
| **DART** | 금융감독원이 운영하는 **전자공시 사이트** (모든 상장사가 공시를 올리는 공식 창구) | 정부가 운영하는 "공시 게시판" |
| **DART OpenAPI** | DART가 제공하는 **자동 조회 창구** (사람이 사이트 들어가는 대신 프로그램이 호출) | 정부 게시판의 "API 자판기" |
| **우리 백엔드(Spring Boot)** | 1분마다 DART에 "새 공시 있어요?"라고 물어보는 우리 서버 | 24시간 일하는 알바생 |
| **PostgreSQL** | 가져온 공시를 보관하는 **데이터베이스** | 우리만의 공시 보관 창고 |
| **종목 마스터(stocks)** | "우리가 관심 가지는 회사 350곳" 명단 (코스피200 + 코스닥150) | 알바생이 들고 다니는 **VIP 명단** |

---

## 1. 큰 그림 — 5단계로 이해하기

```
[1] 관심 회사 명단 만들기  →  [2] 1분마다 DART에 노크  →  [3] 우리 관심사만 골라내기
                                                                ↓
                            [5] 다음 단계(AI 분석)에 알림  ←  [4] 중복 제거 후 DB 저장
```

### 단계별 한 줄 설명

1. **종목 마스터 시드** — "어떤 회사의 공시를 받을지" 명단을 미리 DB에 심어둔다 (1회성).
2. **1분 폴링(polling)** — 우리 서버가 1분마다 DART에 "지난 시간 이후 새 공시 뭐 있어?"라고 자동 조회한다.
3. **필터링** — 받은 공시 중 우리 명단(350곳)에 있는 회사 것만 남긴다. 나머지는 버린다.
4. **중복 제거 + 저장** — 공시마다 고유 번호(`rcept_no`)가 있어, 이미 가진 건 패스. 새 것만 우리 DB에 저장한다.
5. **이벤트 발행** — "새 공시 들어왔어!"라고 다음 팀(AI 분석 모듈)에게 신호를 보낸다.

---

## 2. 각 단계 자세히 (그래도 쉽게)

### 🟦 1단계 — "관심 회사 명단(종목 마스터)" 준비

**왜 필요한가요?**
한국에 상장된 회사는 2,500개가 넘습니다. 우리는 그 중 **시가총액 큰 350개(코스피200 + 코스닥150)**만 봅니다. 작은 회사 공시까지 다 분석하면 비용이 폭증하고, 우리 사용자도 보통 큰 회사 주식을 가지고 있어요.

**어떻게 만드나요?**
- `scripts/data_collection/seed_stocks.py` 라는 파이썬 스크립트를 한 번 실행하면
- KRX(한국거래소)에서 코스피200·코스닥150 명단을 가져오고
- DART에서 각 회사의 "공시용 고유 번호(corp_code)"를 매칭해서
- 우리 DB에 350행을 넣어줍니다 (회사 이름, 종목코드, 시장 구분 등).

> 📁 관련 파일: `scripts/data_collection/seed_stocks.py`
> 📁 결과 저장: DB의 `stocks` 테이블

---

### 🟦 2단계 — "1분마다 노크" (Polling, 폴링)

**폴링이 뭐예요?**
"폴링(polling)"은 우리가 **주기적으로 물어보는 방식**입니다. 마치 알바생이 "혹시 새 거 들어왔나요?"라고 1분에 한 번씩 정부 창구에 물어보는 것과 같아요.

> 📌 왜 1분?
> - 너무 자주 물으면 DART 정부 서버에 민폐 (DART는 하루 호출 횟수 제한이 있음 — 약 20,000회/일)
> - 너무 늦으면 사용자가 뉴스보다 늦게 공시를 받음
> - 1분 = 하루 1,440번 호출 → 한도 안에서 안전

**어떻게 동작하나요?**
- 우리 백엔드의 `DisclosurePollingJob`이라는 스케줄러가 1분마다 자동으로 깨어남
- "마지막으로 성공한 조회 시점부터 지금까지의 공시"를 DART에 요청
- 응답이 오면 그 안의 공시 목록을 다음 단계로 넘김

> 📁 관련 파일: `backend/src/main/java/com/dartcommons/disclosure/DisclosurePollingJob.java`

> 🛡️ 안전장치
> - **재기동 후에도 누락 없이 회복**: "마지막 성공 날짜"를 DB에 저장해두기 때문에, 서버를 잠깐 껐다 켜도 그 사이 공시를 빠짐없이 다시 받아옴.
> - **DART 장애 시에도 멈추지 않음**: 한 번 실패해도 에러만 기록하고 1분 뒤 다시 시도.

---

### 🟦 3단계 — "우리 관심사만 골라내기" (커버 필터)

DART에서 받은 공시에는 **상장사뿐 아니라 비상장 법인, 외국 회사** 등 온갖 게 다 섞여 있어요.

우리 백엔드는:
1. 1단계에서 만든 **관심 회사 350개 명단**을 메모리에 한 번 올려두고
2. 새 공시마다 "이 회사 우리 명단에 있어?" 체크
3. 없으면 **그냥 버림** (저장 안 함)

> 📌 효율적인 이유: 350개를 한 번만 메모리에 로드하면, 100개 공시를 검사할 때 DB를 100번 두드릴 필요 없이 즉시 판별 가능.

---

### 🟦 4단계 — "중복 제거하고 저장하기" (멱등 적재)

**문제 상황**: 1분마다 같은 시간대를 조회하다 보면 **같은 공시를 또 받을 수도** 있어요.

**해결책**:
- 모든 공시에는 **DART가 부여하는 고유 접수번호(`rcept_no`)**가 있어요. (사람의 주민등록번호 같은 것)
- 저장하기 전에 "이 번호의 공시가 이미 우리 DB에 있어?"를 먼저 확인
- 있으면 패스, 없으면 저장 → **절대로 중복 저장이 안 일어남**

> 📌 이 원칙을 어려운 말로 **"멱등성(idempotency)"**이라고 합니다.
> "같은 동작을 여러 번 해도 결과가 한 번 한 것과 같다"는 뜻 — 사용자에게 같은 공시 알림이 2번 가는 사고를 막아주는 핵심 장치.

**저장할 때 같이 하는 일**:
- 공시 제목(`report_nm`)을 보고 **유형을 자동 분류**합니다.
  - 예) "현금배당결정" → `dividend` (배당)
  - 예) "유상증자결정" → `capital_increase` (증자)
  - 예) "단일판매·공급계약체결" → `contract` (계약)
- 이 분류는 **AI가 아니라 규칙(룰)**로 합니다. DART가 준 제목 그대로의 단어를 보고 매칭. (이 단계에서 AI를 쓰면 "환각"으로 잘못 분류할 수 있어 안전한 룰 기반 사용)

> 📁 관련 파일:
> - `backend/.../disclosure/services/DisclosureCollectionService.java` (저장 오케스트레이션)
> - `backend/.../disclosure/services/DisclosureTypeClassifier.java` (유형 분류)

---

### 🟦 5단계 — "다음 단계에 알림 발사" (이벤트 발행)

저장이 끝나면 우리 시스템 안에서 **"새 공시 떴어요! ID는 12345입니다"**라는 신호(이벤트)를 발사합니다.

이 신호를 듣고 있던 **AI 분석 모듈**이 곧바로 일어나서:
- 공시 본문을 가져와
- LLM(인공지능)에게 "이거 호재야 악재야?" 물어보고
- 사용자에게 자연어 요약을 전달하는 다음 흐름이 시작됩니다.

> 📌 이 구조의 장점:
> 수집 모듈과 분석 모듈이 **직접 연결되어 있지 않음** → 한쪽이 느려져도 다른 쪽이 영향받지 않고, 각자 발전시킬 수 있음.

---

## 3. 보너스 — "과거 공시 한꺼번에 가져오기" (백필, Backfill)

서비스 시작 시점에는 우리 DB가 비어 있죠. 그래서 운영자가 한 번:

> "지난 3년치 공시를 한꺼번에 가져와줘"

라는 명령을 내릴 수 있어요. 이걸 **백필(backfill)**이라고 합니다.

- 평소 1분 폴링과 **다른 경로**로 돌아갑니다 (대량 적재용으로 최적화).
- 멱등성 덕분에 중간에 끊겨도 다시 실행하면 안전.
- 단, **AI 분석 트리거는 발사하지 않음** → 과거 9만 건을 한꺼번에 AI에 돌리면 비용 폭탄이라, 별도 배치로 나눠서 분석.

> 📌 현재까지 성과 (2026-06-05 기준): **공시 92,015건 수집 완료** ✅

## 4. 자주 나오는 용어 사전

| 용어 | 쉽게 말하면 |
| ---- | ---- |
| **공시(Disclosure)** | 상장사가 의무적으로 공개하는 회사 소식 (배당, 증자, 계약, 실적 등) |
| **DART** | 정부의 공식 공시 사이트 |
| **OpenAPI** | 사람 대신 프로그램이 데이터를 받아갈 수 있게 만든 창구 |
| **폴링(Polling)** | 주기적으로 "새 거 있어?" 물어보는 방식 |
| **rcept_no** | 공시 1건당 부여되는 고유 번호 (중복 방지용 열쇠) |
| **멱등성** | 같은 동작을 여러 번 해도 결과가 한 번과 같음 (중복/오류 방지) |
| **이벤트(Event)** | 시스템 안에서 "이런 일이 일어났어요"라고 알리는 신호 |
| **종목 마스터** | 우리가 추적하는 회사 명단 (350곳) |
| **백필(Backfill)** | 과거 데이터를 한꺼번에 채워 넣기 |
| **Stage 1** | 우리 분석 5단계 중 1단계 — 수집 + 룰 기반 분류 (AI 안 씀) |

### 📄 코드 ①  — 1분마다 DART에 노크하는 스케줄러

📁 `backend/src/main/java/com/dartcommons/disclosure/DisclosurePollingJob.java`

```java
@Component
@RequiredArgsConstructor
public class DisclosurePollingJob {

    private final DartClient dartClient;
    private final DisclosureCollectionService collectionService;
    private final SystemConfigRepository systemConfigRepository;
```

🔎 **해석**
- `@Component` → "이 클래스는 우리 백엔드가 시작될 때 자동으로 만들어 둬"라는 표시
- 위에서 만든 3개의 부품(DART 호출 담당, 저장 담당, 마지막날짜 기록 담당)을 **장착**한 상태로 시작합니다.

---

```java
    @Scheduled(fixedDelay = 60_000, initialDelay = 10_000)
    public void poll() {
        LocalDate today = LocalDate.now();
        LocalDate from = loadLastPolledDate().orElse(today);
```

🔎 **해석**
- `@Scheduled(fixedDelay = 60_000)` → **6만 밀리초 = 1분마다 자동 실행**하라는 명령. (이게 "1분 폴링"의 정체!)
- `initialDelay = 10_000` → 서버가 켜진 직후 10초만 기다린 뒤 첫 실행 (다른 부품들이 준비될 시간을 줌)
- `today` = 오늘 날짜를 알아냄
- `from` = "지난번에 어디까지 받았더라?" — DB에 기록된 마지막 날짜를 꺼냄. 처음이면 오늘부터.

> 💡 **왜 마지막 날짜를 DB에 저장하나요?**
> 서버를 재시작해도 "어디까지 받았는지" 기억하기 위해서.
> 메모리에만 두면 재시작 = 망각 = 공시 누락 사고로 직결.

---

```java
        try {
            List<RawDisclosureItem> items = dartClient.fetchList(from, today);
            int savedCount = collectionService.collect(items);

            saveLastPolledDate(today);
            log.info("DART polling done: fetched={}, saved={}, window={}~{}",
                     items.size(), savedCount, from, today);

        } catch (DartApiException e) {
            ...
            // lastPolledDate 갱신 안 함 → 다음 폴링에서 동일 윈도우 재시도
        }
    }
```

🔎 **해석**
- `dartClient.fetchList(from, today)` → **DART에 "from부터 today까지 새 공시 줘" 요청**
- `collectionService.collect(items)` → 받은 목록을 **필터/저장 단계**로 넘김 (다음 코드 참고)
- 성공하면 **마지막 날짜를 오늘로 업데이트**
- 실패하면 → 날짜를 **업데이트하지 않음** → 다음 1분 뒤에 **같은 구간 재시도** (자동 회복)

> 💡 이게 바로 "장애가 나도 멈추지 않는다"의 정체. 에러를 잡되, 재시도가 자연스럽게 되도록 설계.

### 📄 코드 ②  — 필터링하고 저장하는 본체

📁 `backend/src/main/java/com/dartcommons/disclosure/services/DisclosureCollectionService.java`

```java
    public int collect(List<RawDisclosureItem> items) {
        Set<String> coveredCodes = stockRepository.findAllStockCodes();

        int savedCount = 0;
        for (RawDisclosureItem item : items) {
            try {
                if (self.collectSingle(item, coveredCodes)) savedCount++;
            } catch (Exception e) {
                log.error("Failed to collect disclosure rceptNo={}", item.rceptNo(), e);
            }
        }
        return savedCount;
    }
```

🔎 **해석**
- `coveredCodes` = "관심 회사 350곳"의 종목코드를 메모리에 **한 번만** 통째로 불러옴
  - (공시 100개를 검사할 때 DB를 100번 두드릴 필요 없음 → 빠름)
- `for (RawDisclosureItem item : items)` → 공시 하나하나에 대해 반복

### 📄 코드 ④  — DART 외부 호출 (방어 장치들)

📁 `backend/src/main/java/com/dartcommons/infrastructure/dart/DartClient.java`

```java
    public List<RawDisclosureItem> fetchList(LocalDate bgnDe, LocalDate endDe) {
        List<RawDisclosureItem> result = new ArrayList<>();
        int pageNo = 1;

        while (true) {
            DartListResponse response = pageFetcher.fetchPage(bgnDe, endDe, pageNo);

            if (response.isNoData()) break;        // 새 공시 없음 → 종료
            if (!response.isOk()) throw ...;       // 에러 응답 → 예외 발생

            result.addAll(response.toItems());

            if (pageNo >= response.totalPage()) break;  // 마지막 페이지 도달
            if (pageNo >= MAX_PAGES) break;             // 안전 상한선
            pageNo++;
        }
        return result;
    }
```

🔎 **해석**
- DART는 한 번에 최대 100건씩만 줍니다. 그래서 **페이지를 넘기며 반복 호출** (1페이지, 2페이지, ...)
- 종료 조건 3가지:
  1. **데이터 없음** → 바로 종료
  2. **마지막 페이지 도달** → 종료
  3. **3,000페이지 초과** → 안전상 강제 종료 (DART가 가끔 같은 페이지를 무한 반복해서 주는 버그가 관측되어 추가된 방어선)

> 💡 외부 시스템은 **신뢰하지 말고 항상 의심하라**가 원칙. 이상 응답에도 무한 루프 안 빠지게 가드.
