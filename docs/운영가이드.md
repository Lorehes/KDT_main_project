# DartCommons 운영가이드

## 1. 인프라 전제

| 항목 | 값 |
|------|-----|
| 클라우드 | AWS EC2 (ap-northeast-2, 서울) |
| 인스턴스 | t3.medium (4GB RAM, ~$30/월) — OpenRouter Cloud LLM 사용 시 |
|  | t3.large (8GB RAM, ~$60/월) — Ollama 온프레미스 LLM 필요 시 |
| OS | Ubuntu 24.04 LTS |
| 스토리지 | 30GB gp3 EBS |
| 보안그룹 | 인바운드 80/443(0.0.0.0/0), 22(내 IP only). **5432·8080·3000 외부 오픈 금지** |
| DNS | CloudFlare → EC2 탄력적 IP (또는 Route53 + ACM) |

## 2. EC2 초기 프로비저닝

### 2-1. 서버 접속

```bash
ssh -i your-key.pem ubuntu@YOUR_EC2_IP
```

### 2-2. Docker + Docker Compose 설치

```bash
# Docker 공식 설치 스크립트
curl -fsSL https://get.docker.com | sudo bash
sudo usermod -aG docker ubuntu
# 재접속 필요 (또는 newgrp docker)

# Docker Compose v2 (Docker 번들 포함 — 별도 설치 불필요)
docker compose version
```

### 2-3. 프로젝트 디렉토리 설정

```bash
mkdir -p ~/dartcommons ~/data/dartcommons-pg
cd ~/dartcommons

# docker-compose.prod.yml을 서버에 복사 (SCp 또는 직접 작성)
# 방법 1: 로컬에서 scp
scp -i your-key.pem docker-compose.prod.yml ubuntu@YOUR_EC2_IP:~/dartcommons/

# 방법 2: 서버에서 직접 작성 (GitHub Raw URL 또는 vi)
curl -fsSL https://raw.githubusercontent.com/YOUR_ORG/dartcommons/main/docker-compose.prod.yml \
  -o docker-compose.prod.yml
```

### 2-4. 환경변수 파일 생성 (서버 직접 — git 미포함)

```bash
cd ~/dartcommons
# .env.prod.example을 참고하여 실제 값 채우기
vi .env.prod
# 권한 제한 (소유자만 읽기)
chmod 600 .env.prod
```

> **보안**: `.env.prod`는 절대 git 커밋 금지. `.gitignore`의 `.env.*` 패턴으로 차단됨.

### 2-5. GHCR 자격증명 영구 설정 (이미지 pull 권한 — 일 1회)

> CI 워크플로에서 GITHUB_TOKEN을 SSH script body에 주입하지 않음 (process list 노출 방지).
> 서버 측 `~/.docker/config.json`에 장기 유효 PAT를 저장하여 CI가 토큰 없이 pull 가능.

```bash
# 1. GitHub → Settings → Developer settings → Personal access tokens (classic)
#    → New token → 권한: read:packages 체크 → 생성 후 복사

# 2. EC2에서 영구 로그인 (token은 ~/.docker/config.json에 저장됨)
echo "ghp_YOUR_PAT" | docker login ghcr.io -u YOUR_GITHUB_USERNAME --password-stdin

# 3. 확인
cat ~/.docker/config.json | python3 -m json.tool | grep -A2 ghcr
```

## 4. Nginx + SSL 설정

### 4-1. Nginx 설치

```bash
sudo apt update && sudo apt install -y nginx
```

### 4-2. 설정 파일 배포

```bash
# nginx/dartcommons.conf를 서버에 복사
sudo cp dartcommons.conf /etc/nginx/sites-available/dartcommons

# your-domain.com을 실제 도메인으로 교체
sudo sed -i 's/your-domain.com/YOUR_DOMAIN/g' /etc/nginx/sites-available/dartcommons

# 활성화
sudo ln -sf /etc/nginx/sites-available/dartcommons /etc/nginx/sites-enabled/dartcommons
sudo rm -f /etc/nginx/sites-enabled/default

# 문법 검증
sudo nginx -t
sudo systemctl reload nginx
```

### 4-3. SSL 인증서 (Let's Encrypt — 권장)

```bash
sudo apt install -y certbot python3-certbot-nginx
sudo certbot --nginx -d YOUR_DOMAIN -d www.YOUR_DOMAIN \
  --email YOUR_EMAIL --agree-tos --no-eff-email
# 자동 갱신 확인
sudo certbot renew --dry-run
```

### 4-4. CloudFlare 사용 시

1. CloudFlare Dashboard → SSL/TLS → Origin Server → Create Certificate
2. 인증서(.pem) + 키(.key) 파일을 EC2 `/etc/nginx/ssl/`에 저장
3. `nginx/dartcommons.conf`의 `ssl_certificate` 경로 갱신
4. CloudFlare DNS → EC2 탄력적 IP A 레코드 등록
5. SSL/TLS 모드: **Full (strict)** 설정

## 7. DB 백업 & 복구

### 수동 백업

```bash
# pg_dump — EC2 ~/backups/ 폴더에 저장
mkdir -p ~/backups
docker exec dartcommons-postgres pg_dump \
  -U dartcommons dartcommons \
  -Fc -f /tmp/dartcommons_$(date +%Y%m%d_%H%M%S).dump

# 컨테이너에서 호스트로 복사
docker cp dartcommons-postgres:/tmp/dartcommons_*.dump ~/backups/
```

### crontab 자동 백업 (일 1회, 새벽 3시)

```bash
crontab -e
# 추가:
0 3 * * * docker exec dartcommons-postgres pg_dump -U dartcommons dartcommons -Fc > ~/backups/dartcommons_$(date +\%Y\%m\%d).dump
# 7일 이상 된 백업 삭제
0 4 * * * find ~/backups -name "*.dump" -mtime +7 -delete
```

### 복구

```bash
# 백업 파일로 복구
docker exec -i dartcommons-postgres pg_restore \
  -U dartcommons -d dartcommons -c /tmp/dartcommons_backup.dump
```

> **중요**: bind mount 경로(`/home/ubuntu/data/dartcommons-pg`)는 Docker 외부 호스트 디렉토리.
> Docker Desktop "Clean / Purge data" 또는 `docker volume prune` 영향을 받지 않음.
> 서버 디스크 포맷/교체 시에만 데이터 손실 위험 → 정기 백업 필수.

## 9. Stage 3 RAG 활성화 & 임베딩 백필 (서버)

> Stage 3 유사 공시 검색을 서버에서 켜는 절차. 코드·컬렉션·백필 잡은 구현 완료 상태
> ([[analysis-stage3-rag-chroma]], [[stage3-embedding-backfill]]) — 남은 것은 인프라 기동 + 백필 1회 실행.
> Ollama는 **임베딩 전용**(nomic-embed-text, ~274MB, CPU 충분) — §8의 챗 LLM용 8GB 요건과 무관, 4GB 인스턴스에서 가동 가능.

### 9-1. 컨테이너 기동 (chroma + ollama)

```bash
cd ~/KDT_mainPJ && git pull
docker compose up -d chroma ollama
# 임베딩 모델 pull (1회, ~274MB)
docker exec dartcommons-ollama ollama pull nomic-embed-text
docker exec dartcommons-ollama ollama list   # nomic-embed-text 확인
```

### 9-2. .env 변경 + 백엔드 재기동

```bash
# .env에 추가/변경 (컨테이너 간 통신 = 서비스명)
CHROMA_ENABLED=true
CHROMA_BASE_URL=http://chroma:8000
EMBEDDING_PROVIDER=ollama
EMBEDDING_BASE_URL=http://ollama:11434
```

```bash
docker compose up -d backend   # env 반영 재기동
# 부트스트랩 확인 — "[ChromaBootstrap] 컬렉션 생성(cosine)" 또는 "이미 존재" 로그
docker compose logs backend | grep ChromaBootstrap
```

### 9-3. 백필 실행 (관리자 API)

> ⚠️ admin API(`/admin/**`)는 nginx가 프록시하지 않아 **외부 URL로 접근 불가**(의도된 미노출 — 2026-07-05 실측 401 확인).
> 서버에서 backend 컨테이너에 직접 호출한다.

```bash
# 시작 — 202 + jobId 반환 (실행 중 중복 시 409)
PW=$(docker exec dartcommons-backend printenv ADMIN_PASSWORD)
docker exec dartcommons-backend wget -qO- --post-data='' \
  "http://admin:$PW@localhost:8080/admin/analysis/embedding-backfill"

# 진행률 조회
docker exec dartcommons-backend wget -qO- \
  "http://admin:$PW@localhost:8080/admin/analysis/embedding-backfill/jobs/{jobId}"
```

- 대상: `content_text` 보유 공시 약 68,000건. 2 vCPU CPU 임베딩 기준 **약 4~7시간** 예상.
- 중단돼도 안전: `last_processed_id` 커서 재개 — 재시작 후 POST 다시 호출하면 이어서 진행.
- 완료 검증: Chroma 벡터 수 확인 (서버 로컬에서만 접근 가능 — 8001은 127.0.0.1 바인드)

```bash
curl -s http://127.0.0.1:8001/api/v1/collections   # id 확인 후
curl -s http://127.0.0.1:8001/api/v1/collections/{id}/count
```

### 9-4. 메모리 주의

- 4GB 인스턴스에서 backend(1g) + frontend(512m) + chroma(512m) + ollama(1.5g) + postgres → 피크 시 빠듯함.
- **백필 중 `docker compose build` 금지** (Gradle/Next 빌드가 RAM 최대 소비원).
- swap 미설정 시 사전 추가 권장: `sudo fallocate -l 2G /swapfile && sudo chmod 600 /swapfile && sudo mkswap /swapfile && sudo swapon /swapfile`
